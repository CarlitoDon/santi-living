# Santi Living i18n Implementation Plan

> **For Hermes:** Execute task-by-task sequentially — each depends on previous.

**Goal:** Enable full Indonesian/English internationalization on `santiliving.com` with locale-aware metadata, components, and sitemap.

**Architecture:** Server components use `getDictionary(locale)` directly (async, server-only). Client components receive translated strings via a `LocaleContext` provider wrapping the layout. Middleware already handles `accept-language` routing.

**Tech Stack:** Next.js 16.2.4, React 19.2.4, no external i18n lib

**Current state:** `src/locales/id.json` & `en.json` only have `common` + `hero` keys. All components hardcoded Indonesian. Sitemap lacks `alternateRefs`.

### Task 2: Create Locale Context + Provider for Client Components

**Objective:** Create a React context so client components (`Header`, `Footer`, `Navigation`) can access locale strings without being server components.

**Files:**
- Create: `apps/web-next/src/contexts/locale.tsx`

**Approach:**
- `LocaleContext` holds `locale: 'id' | 'en'` and `t: Record<string, any>` (the dictionary object)
- `LocaleProvider` receives `locale` and `dictionary` as props from the server layout
- Export `useLocale()` hook for client components
- Export `useT()` hook returning shorthand `t()` function

**Step: Create `contexts/locale.tsx`**

```tsx
'use client';
import { createContext, useContext } from 'react';

type Locale = 'id' | 'en';

interface LocaleContextValue {
  locale: Locale;
  t: (path: string) => string;
}

const LocaleContext = createContext<LocaleContextValue | null>(null);

export function LocaleProvider({
  locale,
  dictionary,
  children,
}: {
  locale: Locale;
  dictionary: Record<string, any>;
  children: React.ReactNode;
}) {
  function t(path: string): string {
    const keys = path.split('.');
    let value: any = dictionary;
    for (const key of keys) {
      if (value == null) return path;
      value = value[key];
    }
    return typeof value === 'string' ? value : path;
  }

  return (
    <LocaleContext.Provider value={{ locale, t }}>
      {children}
    </LocaleContext.Provider>
  );
}

export function useLocale() {
  const ctx = useContext(LocaleContext);
  if (!ctx) throw new Error('useLocale must be used within LocaleProvider');
  return ctx;
}

export function useT() {
  return useLocale().t;
}
```

### Risks & Tradeoffs

- **Next.js 16 quirk:** Need to verify `generateMetadata` with `params` works correctly in this version
- **Client component localization:** Using a context provider adds re-render overhead for locale string lookups — acceptable for this scale
- **SEO impact:** Moving from `/` to `/id/` prefix could temporarily affect rankings if not handled via canonical + hreflang
- **No English content yet:** The English dictionary will be minimal/placeholder — we're building infrastructure, not translating every blog post
